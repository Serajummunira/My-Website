Q-2. System Analyzed: Course Registration Process at UALR

Failure Mode Effects and Analysis (FMEA)

Failure Mode: Student is unable to register for a required course.

Step in Process Potential Failure Mode Potential Causes Effects of Failure Detection Method Severity (1-10) Occurrence (1-10) Detection (1-10) Risk Priority Number (RPN)
Course Registration Unable to register for a required course Prerequisites not met, course full, system error, registration hold Delay in graduation, missing prerequisites Error message, advising support 8 6 5 240

Reasonable vs. Excluded Causes

Reasonable Causes:

  • Prerequisites not met
  • Course is full
  • Registration hold (unpaid fees, missing documents)
  • System error

Excluded Causes:

  • Student forgets to register (This is a user issue, not a system failure)
  • Internet outage (This is an external factor beyond the system’s control)

Suggested Improvements

  • Implement a waitlist system for high-demand courses.
  • Increase system capacity during peak registration times.
  • Send automated prerequisite alerts to students.
  • Enhance error messages to provide clearer guidance.
  • Introduce a **proactive advising notification system** to flag issues early.

Q-3. 5 Whys Technique Analysis

  1. Why couldn't the student register? → The system showed an error message.
  2. Why did the system show an error? → The student did not meet a prerequisite.
  3. Why didn’t the student meet the prerequisite? → The advising system did not notify them.
  4. Why didn’t the advising system notify them? → No automated prerequisite warning.
  5. Why isn’t there an automated warning? → The registration system lacks real-time checks.

Suggested Fixes Based on 5 Whys

  • Implement real-time prerequisite checking.
  • Provide automated alerts if a prerequisite is missing.
  • Require advisor approval for exceptions.

Final Thoughts

The **FMEA analysis** helps identify key failure modes, while the **5 Whys technique** uncovers the root cause. Both approaches highlight the **lack of real-time prerequisite checking** as a major issue. The best way to mitigate this risk is to implement **proactive alerts** and **improve system capacity** to ensure students receive timely updates about their eligibility for courses.

Q-4. Comparison of 5 Whys and Failure Mode Effects Analysis (FMEA)

Applicability and Results

5 Whys Technique

The 5 Whys technique is useful for identifying the root cause of a problem by repeatedly asking "Why?" until the fundamental issue is uncovered. It is a simple, qualitative method that works well for troubleshooting and solving specific, well-defined issues.

For example, if students face difficulties in UALR's course registration system, applying the 5 Whys may reveal that:

  1. Students are unable to register on time. Why?
  2. The registration portal is slow and crashes. Why?
  3. The system cannot handle high traffic. Why?
  4. The infrastructure has limited server capacity. Why?
  5. IT funding for server upgrades is insufficient.

Result: The main issue is insufficient IT funding, leading to poor server performance.

Failure Mode Effects and Analysis (FMEA)

FMEA is a structured, quantitative approach that systematically identifies potential failures, evaluates their severity, likelihood, and detection, and assigns a Risk Priority Number (RPN) to each issue.

For the same course registration issue, FMEA would involve:

  • Failure Mode: System crashes during peak registration.
  • Effects: Delayed registration, frustrated students, administrative burden.
  • Potential Causes: High traffic, outdated infrastructure, software bugs.
  • Detection Methods: Load testing, error logs.
  • Risk Priority Number (RPN): Calculated based on severity, occurrence, and detection ratings.

Result: Provides a structured risk assessment, prioritizing system upgrades and preventive actions.

Comparison

Aspect 5 Whys FMEA
Approach Simple, qualitative, direct questioning. Systematic, quantitative risk assessment.
Use Case Root cause analysis for specific problems. Preventive analysis for potential failures.
Complexity Easy to apply, requires minimal data. Requires data, expertise, and structured evaluation.
Best For Simple issues with clear causes. Complex systems with multiple failure points.

Suitability for Different Problems

5 Whys is best for:

  • Quickly diagnosing operational problems.
  • Addressing human or process errors.
  • Small-scale issues with clear root causes.

FMEA is better suited for:

  • Engineering and technical systems (e.g., software failures).
  • Risk management in high-stakes environments (e.g., healthcare, aviation).
  • Large-scale processes where multiple failures need prioritization.

Conclusion

Both techniques are useful but serve different purposes. 5 Whys is faster and more intuitive, making it ideal for troubleshooting individual problems. FMEA is more structured and effective for large, complex systems that require risk mitigation. For improving UALR’s course registration system, FMEA provides a better long-term solution, while 5 Whys helps in quick diagnostics.
